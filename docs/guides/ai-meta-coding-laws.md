# AI 元编码法则（Thread 规范）

## 1. 目的

本指南定义本 thread 的唯一目标：沉淀“可执行、可索引、可迭代”的项目规范。

这些规范不是说明文，而是对 AI 与人协作开发行为的约束系统（法则之力）。

## 2. 作用域

- 仅在本 thread 产出规范文档。
- 文档统一落地到 `docs/guides/`。
- 每次新增文档，必须在 `AGENTS.md` 增加索引条目。
- 不在本 thread 讨论与规范无关的实现细节。

## 3. 法则写作原则

- 可执行：每条规则必须能被直接执行或检查，避免空泛表达。
- 可验证：每条规则要有“通过/不通过”的判断标准。
- 可追责：规则要明确对象、触发条件、行为要求、例外条件。
- 可演进：规则允许版本迭代，但禁止静默破坏历史含义。

## 4. 单条法则模板（必须）

每条法则建议按以下结构编写：

1. `Rule ID`：唯一标识（如 `LAW-DOC-001`）。
2. `Rule`：一句话约束。
3. `Why`：该约束解决的真实问题。
4. `Trigger`：触发条件（什么时候生效）。
5. `Required Actions`：必须执行的动作。
6. `Anti-patterns`：典型反例（禁止做法）。
7. `Verification`：如何检查是否达标。
8. `Exceptions`：允许例外及审批条件。

## 5. 文档命名与组织

- 文件名使用 kebab-case，体现主题与边界。
- 推荐命名模式：`<domain>-<topic>-laws.md`。
- 一个文档只处理一个核心主题，避免“大杂烩”。
- 交叉主题通过 `Related` 小节互链，不复制内容。

## 6. AGENTS 索引规则

- 在 `AGENTS.md` 新增或维护 `Guides Index` 章节。
- 每篇指南至少包含：标题、路径、1 句适用说明。
- 新增规范文档后，索引更新与文档提交必须同批完成。

## 7. 质量门禁（发布前自检）

发布前必须满足：

1. 文档位于 `docs/guides/`。
2. 已加入 `AGENTS.md` 索引。
3. 每条法则都有可验证标准。
4. 反例与例外清晰，不互相冲突。
5. 术语定义一致，无歧义词。

## 8. 演进机制

- 每次实践后优先修正规则，再修补执行步骤。
- 版本更新时记录：变更动机、影响范围、迁移建议。
- 当旧规则失效时，必须显式废弃（Deprecated），不可直接删除。

## 9. 本 thread 的默认执行约定

- 先产出规范，再进入实现。
- 先明确边界，再补充细节。
- 先定义验证，再定义流程。
