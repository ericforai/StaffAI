# StaffAI: Agent OS 全栈架构设计白皮书 (v1.0)

## 0. 文档概述
本文档详尽记录了 2026-03-28 关于“StaffAI AI 员工管理系统”的 18 层级深度架构设计。StaffAI 被定义为一个企业级 Agent 操作系统（Agent OS），旨在连接 The Agency (144+ 专家库)、DeerFlow (执行引擎) 与企业内部业务系统。

---

## 1. 核心架构：双核管理-执行模型 (Office/Workshop)
系统采用“双核分离”设计，确保管理逻辑的稳定性与执行环境的安全性。

- **管理内核 (Control Plane / "办公室")**：基于 **TypeScript/Express**。负责员工档案、任务编排、权限校验、多级审批流及 Web UI 呈现。
- **执行内核 (Data Plane / "车间")**：基于 **Python/LangGraph (继承 DeerFlow 2.0 理念)**。每个任务运行在独立的 **Docker 物理沙箱**中，支持长时推理与自我修正。
- **同步底座 (Shared State Store)**：采用 **PostgreSQL + Redis + ChromaDB** 共享底座。实现“任务直播”，确保沙箱日志、Thought 流与管理后台实时同步。

## 2. 人工在场：状态挂起与恢复机制 (HITL)
关键操作必须人确认。设计一套“Suspend & Resume”机制：

- **主动询问 (`ask_human`)**：Agent 意识到信息不足或完成草案时，主动挂起任务。
- **上下文回放**：审批界面不仅显示“是/否”，还展示 **Chain of Thought (推导链)**，让人类理解 AI 的动机。
- **指令干预**：审批人可点击“允许”、“拒绝”或“修改建议”，反馈将重新喂给 Agent 触发重思考。

## 3. 治理中心：多级审批与策略引擎 (Governance)
支持从初创到集团级的权限管理。

- **拦截器模型**：分为技能级（主动问）、工具级（敏感拦截如 `rm`）、流量级（Token/时长超限）。
- **多级审批流**：支持 `Step 1 (主管) -> Step 2 (经理) -> Step 3 (合规官)` 的灵活配置。
- **休眠模式**：挂起期间沙箱进入“Hibernation”，保存镜像并释放物理资源，信号到达后唤醒。

## 4. 分层记忆：L1/L2/L3 知识体系 (Layered Memory)
模拟人类员工的知识积累过程。

- **L1 (组织公共)**：公司规章、SOP，对所有 Agent 只读，通过 RAG 注入。
- **L2 (项目共享)**：同项目组成员（人与 AI）共享读写，存储项目特定背景。
- **L3 (私人经验)**：**核心亮点**。记录 Agent 的“避坑指南”、对领导偏好的记忆。作为 **Dynamic System Prompt** 随专家入职注入。

## 5. 结构化大脑：MCP 知识图谱 (Knowledge Graph)
超越简单的向量检索，实现结构化理解。

- **图谱工具集**：提供 `create_entities`, `create_relations`, `add_observations` 等 6 大工具。
- **实体关联**：自动建立 `[人] --(偏好)--> [事]` 和 `[员工] --(擅长)--> [工具]` 的网状联系。
- **入职即老手**：新 Agent 启动时先搜索图谱，瞬间感知组织内部隐性规则。

## 6. 视觉地图：Atlas 可视化视图 (Atlas View)
让“公司大脑”可见、可触。

- **实时生长**：Agent 在沙箱中建立关系时，前端地图实时弹出节点与连线。
- **人机协同编辑**：人类可在 UI 上手动校正、删除错误关系或锁定核心知识节点。
- **全景洞察**：支持按员工、按项目、按属性切换视角，发现公司内部隐藏的连接点。

## 7. 数字总部：三位一体 UI 设计 (Digital HQ)
适配不同角色，跨越“新兴事物”的使用鸿沟。

- **视图 A：交互控制台 (Console)**：类似飞书/Slack，用于@专家下指令，低门槛上手。
- **视图 B：任务看板 (Kanban)**：类似 Jira，带“直播”功能，看卡片流动和执行细节。
- **视图 C：战略控制塔 (Tower)**：Dashboard 视角，看员工地图、知识增量、成本曲线、异常警报。

## 8. 工具安全：侧车保险库架构 (Sidecar Vault)
解决“隔离、性能、安全”的三角难题。

- **核心保险库**：业务密钥（ERP/DB）永远不出“办公室”内核。
- **预热工具池**：MCP Server 运行在独立的子进程池中，隔离崩溃风险，启动毫秒级。
- **影子代理**：沙箱内只有轻量代理，通过 mTLS 隧道向网关请求工具执行，Agent 接触不到密钥。

## 9. 技能演进：动态工具提审流 (Tool Evolution)
允许 AI 员工“发明”工具。

- **自生工具**：Agent 在沙箱中编写 Python 脚本解决新问题。
- **动态提审**：通过 `propose_new_tool` 提交。由“安全专家 Agent”和“人类管理员”双重审核。
- **固化能力**：审核通过后，脚本自动封装为标准 MCP，部署到工具池供全员调用。

## 10. 通信协议：SSE + REST 分离模式
确保“汇报”与“决策”的高效协同。

- **SSE (汇报)**：单向流式回传日志、Thought 和状态变更，实现“任务直播”。
- **REST (决策)**：处理审批、修改建议、中止等原子动作，确保指令的状态一致性。

## 11. 信息载体：统一任务信封 (Task Envelope)
连接 TS 后端与 Python 引擎的唯一协议。

- **包含**：任务标题、人类负责人、AI 承办人、截止日期 (Deadline)。
- **包含**：Agent 基因、上下文包、MCP 工具策略、LangGraph 检查点、成本预算配额。

## 12. 敏捷协作：小组讨论与指挥官机制 (Squad Mode)
1.0 核心协作模式，发挥集体智慧。

- **动态建组**：自动选派“指挥官 (Coordinator)”、“执行者”与“批判者”。
- **指挥官职责**：负责子任务拆解、冲突调解（技术裁决）与阶段性共识总结。
- **语境剪枝**：通过“共识提取”防止讨论区的 Context 爆炸，只保留最有价值的决策记录。

## 13. 并行执行：异步子任务与冲突治理 (Concurrency)
实现真正的研发团队作业效率。

- **子沙箱隔离**：指挥官将任务拆分，开启多个独立子沙箱并行工作。
- **共享白板**：Agent 实时发布中间结论到 Redis 白板，其他沙箱动态感知（如接口定义变更）。
- **合并节点 (`Merge_Node`)**：任务结束后由指挥官进行代码/文档合并，检测并处理逻辑冲突。

## 14. 任务弹性：自我修复与柔性降级 (Resilience)
赋予 AI 组织生命力。

- **原位重启**：检测到沙箱崩溃，自动从检查点重试（限 3 次）。
- **换人接手**：重试失败或表现太差，自动“解雇”当前 Agent，招聘同类专家接手。
- **柔性降级**：非核心分支故障时，隔离该分支，确保其他分支完成并提交“部分成功”的报告。

## 15. 经济合规：预算卫士与核算模型 (Cost-Performance)
确保 AI 员工的经济可预测性。

- **职业定价**：144+ 专家分为基础型 (Flash)、专家型 (Pro)、智囊型 (Opus/o1)。
- **竞标报价**：专家竞标时提交“预估 Token/成本/时长”。
- **熔断机制**：实际消耗超过“维护预算”配额时自动熔断，请人类确认是否追加。

## 16. 人才协议：全渠道入职 (UAD Onboarding)
打造 AI 人才市场。

- **统一定义 (UAD)**：包含基因 (Markdown)、能力 (MCP)、协议 (Internal/External API)。
- **入职编译器**：支持将 144+ Markdown 实时编译为沙箱指令，也支持 API 专家通过“影子沙箱”接入。
- **人才市场**：支持一键安装新的专家组，实现“人才即 App”。

## 17. 智能调度：三段式任务分发 (Dispatcher)
从自动到精准，再到自组织。

- **语义自动匹配**：基于向量检索，为老板自动组建初始专家组。
- **人工精准指派**：老板手动点将，保存专属“铁三角”模板。
- **自由竞标**：**终极形态**。Agent 根据 L3 经验和成功率自主写竞标书，系统择优录用。

## 18. 安全底座：四道数字防火墙 (Security)
StaffAI 的生存底线。

- **第一道 (物理)**：Docker 临时沙箱 + 网络白名单 + 资源 Quota。
- **第二道 (数据)**：Vault 凭证脱敏，Agent 接触不到真实密钥。
- **第三道 (逻辑)**：**红队 Agent 预审**。在老板审批前，安全专家 AI 先对 SQL/代码进行注入攻击检测。
- **第四道 (治理)**：多级审批门禁 + 一键全线熔断按钮 + 不可篡改的黑匣子审计日志。

## 附录：V1.1 逻辑优化与安全增强补丁 (Revision Notes)

基于架构首轮审计，特补充以下 4 条修正案，以解决执行层与管理层的潜在逻辑缺陷：

### 1. [工程] 并行协作的“强制中断”机制 (Blackboard Sync)
- **缺陷**：异步并行模式下，子沙箱可能存在状态同步延迟（白板竞态）。
- **修正**：在“共享白板”中引入 **“广播中断 (Broadcast Interruption)”**。当指挥官发布核心变更（如 API 定义修改）时，所有关联子沙箱强制触发一次 `Mini-rethink`，确保 Agent 在继续编码前已感知最新上下文。

### 2. [产品] 审批负担的“摘要降维”设计 (Cognitive Relief)
- **缺陷**：多级汇总审批可能导致人类管理者的认知负荷过重，造成“盲目通过”。
- **修正**：要求 **指挥官 Agent** 必须提供 **“人类语言意图摘要”**。审批界面将复杂的技术推导链折叠，优先展示大白话版本的“行动计划书”与“风险评估建议”。

### 3. [业务] 人才市场的“冷启动补贴”算法 (Diversity Bias)
- **缺陷**：竞标机制可能导致成功率高的老专家垄断任务，新专家无法获得 L3 经验。
- **修正**：在调度算法中引入 **“多样性加权 (Diversity Weight)”**。系统强制为新入职或低使用率的专家提供一定比例的“试错配额”，确保人才梯队的持续进化与多样性。

### 4. [安全] 审计防御的“跨模型验证”原则 (Cross-Model Audit)
- **缺陷**：若审计 AI 与执行 AI 使用同一模型家族（如均为 GPT-4），可能存在共模失效，无法发现逻辑漏洞。
- **修正**：强制执行 **“跨模型审计协议”**。执行 Agent 产出的代码/SQL，必须由不同模型家族（如 Claude 审 GPT，或 Gemini 审 Claude）的红队 Agent 进行预审，以最大化拦截逻辑死角。

## 20. 工程实施细节与架构决策 (V1.2 - 2026-03-28 Review)

基于首轮工程评审 (Engineering Review)，特锁定以下 8 项核心技术实现方案，确保 StaffAI 1.0 的高可用与高性能：

### 20.1 架构与稳定性 (Architecture & Reliability)
1.  **Checkpoint 状态恢复逻辑**：在 `Task_Envelope` 中内置原子状态检查点。当 SSE 连接意外中断时，TS 网关支持自动拉起新沙箱并从上一个 Checkpoint 断点续传，确保长时任务的可靠性。
2.  **异步非阻塞 MCP Proxy 队列**：在 TS 层实现带并发控制和速率限制的工具调用网关。防止高频工具请求阻塞 Node.js 事件循环，确保 Digital HQ 实时界面的丝滑响应。
3.  **知识图谱写入串行化锁 (Sequential Lock)**：在图谱更新工具中增加 `AsyncMutex` 锁机制。针对多 Agent 并行写入场景，强制实施原子操作，防止 SQLite 数据库锁定或关系图谱数据冲突。

### 20.2 代码质量与模块化 (Code Quality & Modularity)
4.  **Service-Adapter 解耦模式**：严禁将业务逻辑散落在 API 路由中。采用 `StaffAIGateway` (领域服务) + `SandboxAdapter` (双核协议适配) 的标准分层架构，提升代码的可测试性。
5.  **统一 Agent 注册中心 (Registry)**：封装 `AgentRegistry` 单例，直接对接 `The Agency` 现有的 Scanner 逻辑。提供统一的 UAD 基因编译接口，严禁在业务代码中直接硬编码读取专家 Markdown 文件。
6.  **强类型错误治理 (Error Governance)**：定义全局 `StaffAIError` 体系。针对沙箱启动失败、SSE 协议错误、工具越权等场景，实施分级捕获与 UI 友好回传，杜绝“全平台崩溃”隐患。

### 20.3 性能优化策略 (Performance & Efficiency)
7.  **异步预加载与二级缓存 (Prefetching & Caching)**：在用户打开“创建任务”页面时，预加载 L1/L3 常用知识片段。在 TS 内存中缓存热点记忆，将任务启动耗时从“秒级”降至“毫秒级”。
8.  **SSE 缓冲管理与主动回收 (Flow Control)**：限制单个任务流的缓冲区大小。设定任务空闲超时阈值，超时自动断开 SSE 并将沙箱置入“休眠状态 (Hibernation)”，确保持久运行下的内存安全。

## 19. 演示场景 (Demo Scenarios)

为了验证 1.0 架构的实战价值，系统将首批预置以下三个核心业务场景的“专家小组模板”：

### 场景一：全球化跨境电商团队 (Cross-border E-commerce Squad)
*   **目标**：实现从选品、文案生成到海外客服的自动化闭环。
*   **专家构成**：
    *   `Global Product Researcher` (选品专家)
    *   `TikTok Marketing Strategist` (营销策略)
    *   `Visual Storyteller` (视觉叙事)
    *   `Customer Support Responder` (多语言客服)
*   **架构验证点**：多语言知识库 (L1/L2)、多 Agent 协作 (Squad Mode)、图谱中的“用户偏好”记录。

### 场景二：全自动软件开发与运维组 (DevOps & Engineering Squad)
*   **目标**：从需求文档直接生成代码、执行测试并自动化部署。
*   **专家构成**：
    *   `Software Architect` (架构师/指挥官)
    *   `Backend Developer` (后端专家)
    *   `Security Engineer` (安全审计)
    *   `SRE / DevOps Automator` (运维专家)
*   **架构验证点**：Docker 物理沙箱隔离、工具级敏感拦截 (L2)、异步并行开发与合并节点 (Merge Node)。

### 场景三：深度研究与战略咨询机构 (Deep Research & Consulting Squad)
*   **目标**：针对复杂商业问题进行全网搜索、多维分析并产出专业报告。
*   **专家构成**：
    *   `Trend Researcher` (趋势研究)
    *   `Financial Analyst` (财务分析)
    *   `Senior Project Manager` (主笔/指挥官)
*   **架构验证点**：SSE 流式推导链直播、分层记忆中心 (L1/L2/L3)、内部核算模型 (针对高昂 Token 的 ROI 分析)。
