## 1. Goal（本轨道总目标）

在 **不破坏单条交付主线** 的前提下，为 StaffAI HQ 增加三类 **可观测、可治理、可迭代** 的平台能力，对应 `AGENTS.md` 路线图未勾选项：

| 路线图项 | 本 spec 称呼 |
|----------|----------------|
| Real-time Cost/Token Analytics Dash | **成本 / Token 看板** |
| Multi-turn Parallel Autonomy (L3) | **L3 可控并行** |
| Cross-Agent Shared Memory (L2) Graph | **L2 跨 Agent 共享记忆（图）** |

三者 **可分期交付**，但共享同一套 **治理与可观测** 底座（审计、配额、显式用户同意），避免三套独立烟囱。

---

## 2. North-Star（用户可感知价值）

> 组织在 **看清 AI 消耗** 的前提下，对 **多路并行执行** 设定上限与审批，并让 **跨任务 / 跨 Agent 的有效经验** 以可控方式回流到后续交付，而不把操作者拉回「控制台迷宫」。

与 Lighthouse 的关系：**Lighthouse 回答「现在该做什么」**；本轨道回答 **「花了多少、并行是否安全、别人已经知道了什么」** —— 默认 **二级入口或折叠模块**，不抢占 Hero。

---

## 3. Workstream A — 成本 / Token 看板

### 3.1 Problem

缺乏 **任务 / 执行 / 模型调用** 维度的实时或准实时成本与 token 聚合，无法做预算告警、选型复盘与治理审计。

### 3.2 Scope（MVP → 增强）

**MVP（建议首版）**

- **数据源**：以现有或规划中的 **执行记录、runtime 适配器回传** 为准（如 `execution` trace、`costLogs`、各 adapter 统一字段）；若当前缺失，在实现计划中增加 **单一写入路径**（禁止各 UI 各算一套）。  
- **聚合维度**（至少其二）：按 **任务**、按 **时间窗口（日/周）**、按 **执行器/模型别名**（若可解析）。  
- **展示**：HQ 内 **独立页面或「设置 / 运营」下钻页**（**非** Lighthouse Hero 区域）；支持表格 + 简单趋势（折线/柱状即可）。  
- **刷新**：准实时（轮询 30s～2min 可配置）或 SSE/WebSocket 二选一，在 plan 阶段选定。

**增强（后续版本）**

- 组织 / 项目级 **预算阈值** 与 **只读告警**（邮件/Slack 可接 webhook）。  
- 导出 CSV；与 **审批 / 高风险执行** 联动展示（「本任务因何多花」）。

### 3.3 Non-Goals（首版不做）

- 替代云厂商账单对账（财务级精确度）。  
- 在 **未统一 token/cost 字段** 前做大屏特效。  
- 将成本数字塞进 **每一步向导** 主 CTA（避免打断漏斗）。

### 3.4 Success Criteria

1. 对 **至少一条** 真实或 E2E mock 执行链路，看板可显示 **可解释** 的 token/cost 汇总（单位与口径在 UI 脚注说明）。  
2. 新增/变更字段 **有 schema 或类型定义**（前后端对齐）。  
3. **性能**：列表页在常规数据量下可交互（具体阈值在 plan 中写死）。

## 4. Workstream B — L3 可控并行（Multi-turn Parallel Autonomy）

### 4.1 Problem

多 Agent / 多子任务并行时，易出现 **竞态、重复劳动、审批风暴、成本失控**，与「企业治理」目标冲突。

### 4.2 Scope（「可控」定义）

**能力描述**：在 **明确边界** 内允许多个 execution / subagent **并行推进**，且系统始终满足：

1. **策略开关**：按任务类型、风险等级、组织策略 — **默认保守**（例如先串行，显式开启「受控并行」）。  
2. **并发上限**：全局 / 每任务 / 每用户 **硬顶**（数字可配置）。  
3. **审批与工具门**：高风险工具调用 **不因并行而绕过** 现有 HITL；并行分支 **独立 checkpoint**。  
4. **可观测**：并行拓扑在 UI 或日志中 **可追踪**（父任务 → 子执行 ID），便于排障。

### 4.3 Non-Goals（首版不做）

- 任意无上限自主分叉（「完全自主 L3」）。  
- 与 L2 记忆图 **强绑定**（可先并行交付，后续再接记忆注入策略）。

### 4.4 Success Criteria

1. 在 **测试环境** 可复现：同一任务下 **2 路并行** 执行，治理规则仍拦截应拦截的工具调用。  
2. 超出并发上限时 **明确拒绝或排队**，并有用户可见说明。  
3. 文档化 **与 Lighthouse 的交互**：并行开启入口 **不在** 向导 Step1～3 的默认路径上（例如任务详情「高级」或策略面板）。

## 5. Workstream C — L2 跨 Agent 共享记忆（图）

### 5.1 Problem

知识停留在 **单次任务 / 单次会话**，优质澄清、计划模式、踩坑记录 **无法结构化复用**；纯文本 `company_knowledge` 式列表难以表达 **实体关系与置信度**。

### 5.2 Scope（概念模型 · 实现可迭代）

**图语义（逻辑层）**

- **节点类型（示例）**：`Task`、`Intent`、`Artifact`、`Decision`、`AgentRun`、`Concept`（具体集合在 plan 中收敛）。  
- **边**：派生、反驳、复用、引用、审批结论等 **有类型** 的关系。  
- **写入**：默认 **异步摘要 + 人工确认** 或 **高置信自动边**（策略可配）；禁止静默覆盖关键事实。  
- **读取**：对 **新任务 / 新意图** 的检索 API：**top-k 子图** 或 **向量 + 图遍历** 混合（实现阶段选型）。

**与产品 UI**

- 首版可为 **任务详情侧栏「相关记忆」** 或 **调试视图**，不进入 Lighthouse 首屏。  
- 与 **模板飞轮** 衔接：模板实例化时可 **可选注入** 关联子图（策略在 plan 定义）。

### 5.3 Non-Goals（首版不做）

- 全库知识图谱可视化性能极限（百万边大屏）。  
- 跨租户共享（默认 **组织内** 隔离）。  
- 自动学习 **未经审计** 的敏感业务数据（须符合现有合规与 host policy）。

### 5.4 Success Criteria

1. **写入路径单一**：至少一种来源（如「任务完成」或「存模板」）能生成 **可查询** 的图记录。  
2. **读取路径**：新任务创建或意图澄清时，可 **拉取并展示** 若干条相关记忆（哪怕首版为 debug JSON）。  
3. **权限**：与现有 `x-user-id` / 组织模型 **对齐**，无越权读取。

## 6. Cross-cutting — 与 Lighthouse 共存的 UX 原则

1. **主路径单一**：`/tasks` Hero **只服务**「当前交付焦点」，成本/并行/记忆 **不占用 Hero 主 CTA 位**。  
2. **渐进披露**：高级能力放在 **二级导航、任务详情、设置页**。  
3. **同一套可观测**：`delivery-analytics` 类事件 **可扩展** 到成本/并行/记忆（字段名在实现 plan 中登记），避免三套日志格式。  
4. **版本节奏**：建议 **A（成本）→ B（并行）或 C（记忆）** 二选一为第二里程碑（由依赖决定：若有统一 execution 元数据，成本优先；若治理要先于放量，并行策略优先）。

## 7. Dependencies & Risks

| 依赖 / 风险 | 缓解 |
|-------------|------|
| Execution / adapter **未统一** cost/token 字段 | Workstream A 第一期以 **schema + mock + 单 adapter 打通** 为门禁，再铺全适配器。 |
| L3 并行与审批竞态 | 单写者队列或 per-branch 锁；E2E 覆盖「并行 + 拦截」场景。 |
| L2 图与现有 `company_knowledge` / memory API 重复 | spec 层明确 **迁移或双写期**；plan 中定废弃路径。 |
| 主路径再次碎片化 | **本 spec 门禁** + UX 原则第 1～2 条；每个 workstream **独立 PR 可回滚**。 |
