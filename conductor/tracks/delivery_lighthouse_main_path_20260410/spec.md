# Specification: Delivery Lighthouse — 端到端主路径与完成定义

## 1. Goal

将「从模糊想法到可验收交付」收敛为 **一条可讲清楚、可走完、可失败可恢复** 的主用户路径；主界面以 **当前进行中的一件事 + 明确下一步 + 阻塞原因** 为先，降低「控制台/dashboard 堆砌」带来的认知负担。

本 spec 定义 **产品行为与验收标准**；实现任务见同目录 `plan.md`。

## 2. North-Star User Story（单一成功故事）

> 用户输入一句需求 → 与系统对齐成可确认的规格 → 看到实施计划 → 选定负责人并创建任务 → 执行（含必要时审批）→ 在任务内获得可下载或可外链的产出物 → 一键将本次路径存为模板以便复用。

该故事在 UI 上必须 **可按顺序完成**，且每一步有 **完成态** 与 **失败时的可操作出口**。

## 3. Journey Stages — 完成定义与失败恢复

以下阶段与现有实现对齐关系（便于落地，非最终命名约束）：

| 阶段 ID | 用户感知名称 | 典型入口 | **完成态（Done）** | **常见失败** | **用户可执行恢复** |
|--------|-------------|----------|-------------------|-------------|-------------------|
| **S0** | 选择如何开始 | `/tasks` | 用户进入「高级向导」或明确选择「简易任务」 | — | 随时切换模式（不丢已保存草稿则更佳） |
| **S1** | 输入原始需求 | 向导 Step 1（无 draft） | 成功创建 intent，`intentId` 已生成并可继续澄清 | 网络错误、校验失败、API 4xx/5xx | **重试**提交；修改输入后重试；**取消**回 S0 |
| **S2** | 澄清成规格 | 向导 Step 1（有 draft） | 后端/产品定义的「可进入设计确认」条件满足（例如澄清轮次或状态字段达标）；用户可见「进入下一步」类 CTA | 流式中断、模型错误、超时 | **重试**发送；**跳过澄清**（若产品允许：带风险说明并记日志）；**返回**修改原始描述（若支持） |
| **S3** | 确认设计方案 | 向导 Step 2 | 用户确认 design；状态锁定设计摘要 | 确认失败 | **重试**确认；**返回**澄清（若状态机允许） |
| **S4** | 实施计划预览 | 向导 Step 3 | 用户可见计划正文且可发起「创建任务」 | 计划生成失败 | **重新生成计划**（若 API 支持）；否则 **重试**；**人工编辑**（后续迭代，本阶段可标为 P2） |
| **S5** | 正式任务与负责人 | 任务详情 / 创建后跳转 | `TaskRecord` 存在，`assignee` 符合业务规则（若必填则已选） | 创建任务失败 | **重试**创建；**改参数**（负责人/优先级）后重试 |
| **S6** | 执行与治理 | 任务详情 + 审批队列 | 执行进入终态之一：`completed` / `failed` / `cancelled`；若需审批则 `waiting_approval` 时有明确审批入口 | 执行失败、审批超时 | **重新执行**（若允许）；**去审批**；查看 **阻塞原因**（风险/工具拦截文案） |
| **S7** | 可验收产出物 | 任务详情「产出/Artifacts」 | 至少一种可验收形态：**可下载文件**、**可点击外链**、或 **结构化摘要**（三选一作为 MVP；多种并存更佳） | 无产出、链接失效 | **刷新**；**重新跑执行**；联系管理员（外链权限类，仅文案） |
| **S8** | 存为模板 | 任务详情 | 用户触发「存为模板」且成功；模板中心可检索到（若已实现 Template Center） | 保存失败 | **重试**；检查标题/权限提示 |

**完成定义（整故事 E2E）**：从 S1 到 S7 在「快乐路径」上可走通；在 **至少一种** 注入失败场景下，用户无需开发者工具即可完成表中「恢复」动作之一并到达下一合法状态。

## 4. UI Contract — 减少「控制台感」

### 4.1 主路径 Hero 区域（优先级 P0）

在 **`/tasks`（任务主工作台）** 顶部或首屏固定区域提供 **Delivery Lighthouse** 模块：

1. **当前焦点（Current）**  
   - 一句人话描述：例如「正在：完善需求 — 向导第 2 步」或「正在执行：{任务标题}」  
   - 数据可来自：进行中的 `intentId` 草稿、未完成任务中优先级最高的一条、或用户最近操作的一条（规则需在实现时在 `plan.md` 子任务中写死并测）

2. **主行动（Primary CTA）**  
   - 唯一主按钮：例如「继续」「去审批」「查看产出」  
   - 点击目标必须与当前焦点一致，避免跳转后用户迷路

3. **阻塞说明（Blocker）**  
   - 若无阻塞：显示「无阻塞」或隐藏该行  
   - 若有：展示 **原因摘要** + 可选 **次要操作**（如「重试」「查看详情」）

### 4.2 信息架构原则

- **默认折叠或弱化**：并发的多面板摘要（多任务统计、次要 execution 卡片）不得抢占 Hero 的视觉层级。  
- **向导内**：每一步展示 **步骤条或阶段标签** + 本步 **完成条件提示**（可与后端状态字段映射）。  
- **术语**：对用户优先使用「需求 / 计划 / 任务 / 产出 / 模板」；内部状态名（`waiting_approval` 等）仅作开发映射表。

## 6. Non-Goals（本轨道首版不做）

- 完整 OKR / 多项目组合视图  
- 实时成本大屏（可与后续「成本/token」轨道衔接）  
- 全自动跳过所有人工确认（与治理目标冲突）  
- 移动端专用布局（桌面优先）

## 7. Success Criteria（验收）

1. **文档**：本 `spec.md` 与 `plan.md` 评审通过（产品 + 前端负责人可指着阶段表走查）。  
2. **行为**：Happy path 从原始输入到任务内可见产出物可走通（在已有后端能力范围内；缺失 API 则在 `plan.md` 拆出后端子任务）。  
3. **失败路径**：至少覆盖 **S1 创建 intent 失败** 与 **S6 审批阻塞** 两类，UI 上可见恢复动作且可点击。  
4. **Hero**：在 `/tasks` 存在 Current + Primary CTA + Blocker 三要素；无进行中任务时有合理的空状态 CTA（如「发起新交付」）。  
5. **测试**：新增或更新 Playwright 用例覆盖 Hero 可见性与向导一步完成提示（与现有 `tests/e2e` 风格一致）；关键纯逻辑可抽 small unit test（可选）。

## 8. Resolved decisions（2026-04-10）

1. **焦点仲裁（简易模式 /tasks）**：`deriveDeliveryFocus` 优先级为 — **待处理审批（全队列）** → **执行中任务** → **`waiting_approval` 任务** → **其他未完成（按 `updatedAt` 最新）** → **空闲**。高级向导模式下 **始终以向导进度为 Hero 焦点**（审批不抢占，避免打断漏斗）。  
2. **跳过澄清**：**首版不做**；仅提供澄清失败后的 **重试**（非流式 `sendMessage` 重发上一句）。  
3. **产出物 MVP**：任务详情 **Assignments → Artifacts** 提供 **结构化正文**；`ArtifactsPanel` 另支持 **复制全文、单条/合并导出 Markdown**（满足 spec §3 **S7**「结构化摘要」及可带走的导出形态）。外链/独立下载文件列为后续迭代。
